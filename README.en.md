# Nilo

> Japanese version: [README.md](README.md)
>
> The Japanese README is the primary document. This English README is a supplementary introduction.

Nilo is a local CLI for recording the **current state, completion criteria, verification results, review results, and human acceptance decisions** of development work delegated to AI agents.

Its guiding principle is **Evidence Before Trust**. Nilo does not treat an AI agent's "done" as final by itself; it keeps the change, verification, review, and human acceptance as separate records.

Nilo is not a task-management SaaS, CI system, or security boundary. It assumes cooperative AI agents using the CLI / MCP path, and stores the evidence that would otherwise disappear into chat history in `.nilo/nilo.db`.

## Why Nilo Exists

When you develop with AI agents, it can become unclear what is actually done, whether verification really passed, and whether another agent can pick up the same context.

Nilo keeps the work intent, execution results, verification, review, and unresolved items as project state. Humans do not need to memorize every command; they can ask an AI agent in natural language and inspect the evidence Nilo records.

## What Nilo Does

- Shows where AI-assisted work currently stands
- Records completion criteria for each task
- Records verification commands and results
- Keeps review results and unresolved findings visible
- Ties verification, review, and completion decisions to git snapshots
- Separates AI work reports from human acceptance decisions
- Preserves failure logs, policy notes, and follow-up work
- Reuses common workflows as recipes
- Creates verified backups of the Nilo DB

## Install

Requirements:

- Python 3.12 or later
- Git

```bash
git clone https://github.com/endow/nilo.git
cd nilo
python -m pip install -e .
```

Check the install:

```bash
nilo --version
nilo status
```

## Fastest Start

Initialize Nilo once at the root of your project.

```bash
nilo init
nilo work "shorten the README" --intent change
```

This mainly creates `.nilo/nilo.db`, runtime instructions for AI agents, and local exclude settings. The generated runtime files are usually not committed.

The repository-level `AGENTS.md` and `CLAUDE.md` files are intentionally empty placeholders so they do not conflict with the runtime instructions generated by each user's `nilo init`. Agents should follow the local generated instruction files in the project where they are working.

For normal work, use `nilo work` as the entrypoint and declare `--intent change`; use `--intent inspect` for read-only requests. Nilo can bundle task creation, recipe selection, verification recording, and AI completion reporting behind a change session.

After that, humans usually do not need to type detailed Nilo commands. Ask your AI agent as you normally would.

```text
Check Nilo's state before you start.
```

```text
Record the verification result and unresolved items before reporting back.
```

```text
Look at Nilo's evidence and tell me whether this is ready to accept.
```

The agent reads Nilo, follows the current instruction, handles completion criteria and verification, and reports back. The human decides whether to accept, send back, or ask for more checking.

## Verification Levels

Nilo is not a way to skip verification. It records the level of verification that fits the stage of work.

- `changed check`: a fast in-progress check that selects shards from changed files.
- `smoke` / `compat`: short checks for basic CLI behavior and compatibility.
- `full check`: the full-shard equivalent used for releases or broad changes.
- `audit snapshot`: a stricter check that evidence and completion decisions still match the current git state.

`release prepare` may defer the full check until publish time when it is safe to do so. `release publish` always requires a valid full check before public operations; if one is missing, Nilo runs it first.

## Natural-Language Prompts

```text
What's next?
```

```text
Is anything still unresolved?
```

```text
Did verification pass?
```

A useful answer should look more like this than a bare "done":

```text
The task is waiting for review.
Verification passed.
There is 1 unresolved finding.
Next step: address that finding.
```

## `nilo view`

```bash
nilo view
```

`nilo view` opens a read-only local browser view for tasks, verification, reviews, failure logs, and task analysis. 

## Recipes / Roadmaps / Overdrive

Recipes start common work in a consistent way. Documentation updates, bug fixes, performance investigations, and release preparation can become normal Nilo tasks with completion criteria and verification expectations.

Roadmaps keep large work from becoming one unchecked implementation step. They capture goals, non-goals, and success criteria first, then split accepted plans into executable tasks after human approval.

Overdrive continuously advances AI-agent work along accepted roadmap items. It does not remove final human judgment, and it stops for destructive operations, external publication, secret access, ambiguous specifications, and similar safety gates.

## MCP Integration

Nilo is not tied to one AI product. Codex, Claude Code, ChatGPT, local LLMs, and other agents can use Nilo through the CLI or MCP (Model Context Protocol).

Being able to call an MCP tool does not prove it is looking at the current repository's Nilo DB. In multi-workspace use, agents should check identity and avoid using evidence from the wrong repository.
